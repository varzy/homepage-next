# PRD 1.0 — Notion 内容增量同步

## 背景

本站内容托管在 Notion，通过一套自定义脚本将 Notion 数据库内容拉取到本地 Markdown 文件，再经 Next.js 构建成静态站点。目前通过 GitHub Actions 每 20 分钟定时触发一次完整的内容同步（`fetch:all`），同步后自动执行检查、提交并推送，触发重新部署。

其中 kotoba 数据库增长最快，预计每天至少新增一条记录，长期来看条目数量将达到数千条。

---

## 问题

### 问题一：每次运行都全量拉取元数据

当前 `fetch:all` 每次执行时，无论 Notion 内容是否有变化，都会向 API 请求整个数据库的所有条目元数据（分页拉取，每页 100 条）。条目越多，无效的 API 调用越多，执行时间越长。

### 问题二：`--force` 在规模扩大后不可用

`--force` 模式会对数据库中的所有条目逐一执行完整处理（拉取页面内容、转换 MDX、上传图片），条目数量增多后极容易触发 Notion API rate limit，且耗时难以接受。

---

## 目标

1. 日常同步只请求"自上次运行以来有变动"的条目，API 调用量接近 O(1)，不随总条目数增长
2. 保留完整同步能力，用于定期清理孤儿文件、确保本地与 Notion 一致性
3. 上述两种模式均与 GitHub Actions 定时任务兼容，同步状态可在 CI 环境中跨次持久化
4. 现有命令行接口向后兼容（`--force` 继续有效）

---

## 非目标

- 不引入外部状态存储（数据库、对象存储等）
- 不改变内容文件本身的格式或结构
- 不修改 Next.js 构建侧的任何逻辑

---

## 需求

### R1 — 增量模式（默认）

- 脚本默认以增量模式运行，无需额外参数
- 增量模式下，Notion 查询仅返回 `last_edited_time` 晚于上次成功运行时间的条目
- 增量模式不执行孤儿文件清理（因为无法得知全量删除情况）
- 首次运行（无历史状态）时，自动降级为完整同步

### R2 — 完整同步模式

- 通过 `--full-sync` 参数触发
- 全量拉取所有条目元数据，执行孤儿文件清理
- 内容处理仍保留增量判断（只处理实际有变化的条目）

### R3 — 状态持久化

- 同步状态（上次成功运行时间）存储于项目根目录的 `.fetch-state.json` 文件中
- 该文件提交到 git，确保 GitHub Actions 每次 checkout 后状态可用
- 只有在本次运行确实产生内容变化时才更新并提交状态文件，避免产生无意义的空提交

### R4 — GitHub Actions 集成

- 现有的 `scheduled-release` workflow（每 20 分钟）继续使用增量模式（`fetch:all`，即默认行为，无需修改）
- 新增 `scheduled-full-sync` workflow，每月 1 日和 16 日（北京时间上午 10:00）自动执行完整同步，用于孤儿文件清理

### R5 — 向后兼容

- `--force` 参数继续有效，行为与现在一致（强制全量处理所有条目）
- 现有的 `fetch:all`、`fetch:kotoba` 等命令无需加参数即可正常工作

---

## 成功标准

- 在 Notion 内容无变化时，`fetch:all` 的 API 调用次数不随数据库条目数增长
- GitHub Actions 在无内容变化时不产生新的提交
- 每月两次的完整同步能正确清理已从 Notion 删除的本地孤儿文件
- 所有现有测试继续通过

---

## 开放问题

- **`dataSources.query` 是否支持 `last_edited_time` 时间戳过滤器？** 当前代码使用的是非标准 API，需要在实施前验证该过滤器是否兼容。若不兼容，增量查询过滤需改用其他字段（如 `published_time`），效果退化为方案 A。
